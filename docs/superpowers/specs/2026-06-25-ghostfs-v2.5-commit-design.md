# ghostfs v2.5 — Atomowy commit + crash-consistency: design

**Data:** 2026-06-25
**Część:** v2 (CoW B-tree). Domyka poprawność: formalizuje commit + DOWODZI atomowości sweepem
awarii (analog v1 B/test_crash). Korona poprawności v2.

## Model commitu (CoW, atomowy — formalizacja istniejącego gh2_fs_commit)
Operacje mutują `fs->fs_root` w pamięci (CoW, nowe bloki alokowane natychmiast, stare → defer-
free). Trwałość przez `gh2_fs_commit`:
1. **Faza 1 — bariera danych:** `fsync(dev->fd)` — wszystkie nowe bloki (węzły drzewa + bloki
   danych, zapisane przez gh_disk_write do cache OS) trafiają trwale na dysk PRZED superblokiem.
2. **Faza 2 — atomowa podmiana korzenia:** `gh2_commit_super` — zapisz superblok w slot
   `gen&1` z `root_tree=fs_root` + `next_ino` + `gen+1` + csum + read-back verify; fsync. To
   JEDYNY punkt atomowy (ping-pong v2.0; nigdy nie nadpisuje żywego slotu).
3. **Faza 3 — zwolnienie starych:** `gh2_txn_alloc_commit` — oznacz defer-free jako wolne
   (teraz nieosiągalne; tylko w pamięci, mark-sweep przy mount i tak odbuduje).

**Atomowość (niezmiennik):** superblok pisany OSTATNI, po trwałości wszystkich nowych bloków.
- Awaria przed fazą 2 → stary superblok ważny → STARY spójny stan (nowe bloki nieosiągalne).
- Awaria w trakcie fazy 2 (rozdarcie SB) → drugi slot (stary) ważny → STARY stan.
- Awaria po fazie 2 → NOWY trwały stan.
Zawsze stan STARY albo NOWY — nigdy częściowy/uszkodzony.

## Fault injection — pokrycie zapisu superbloku
`gh_disk_write` ma już `dev->fail_after` (failuje N-ty zapis). Zapis superbloku idzie surowym
`pwrite` (gh2_super.c) — **dodaj ten sam check `dev->fail_after`** w `sb_write_slot`, by sweep
pokrywał też punkt podmiany korzenia. Wtedy jeden licznik fail_after pokrywa KAŻDY zapis commitu
(węzły, dane, SB).

## gh2_fsck (formalizacja walidatora z testów)
`int gh2_fsck(struct gh2_fs*, int *issues)` — weryfikuje (zwraca liczbę niespójności):
- każdy INODE_ITEM osiągalny z roota (DFS po katalogach) — brak sierot;
- każdy DIR_ITEM → istniejący INODE_ITEM docelowy;
- nlink i-węzła == liczba dowiązań (katalog: 2 + podkatalogi);
- każdy disk_block ekstentu: w granicach, oznaczony w mapie, UNIKALNY (brak współdzielenia
  między ekstentami/i-węzłami), nie koliduje z węzłem drzewa;
- size i-węzła spójne z ekstentami (brak ekstentu poza size po truncate);
- mapa == świeży mark-sweep (po mount tautologia, ale sprawdza book-keeping).
(Read-only; nie naprawia — repair to przyszłość.)

## Testy — crash-consistency sweep (BRAMKA, `tests/test_v2crash.c`)
1. **Sweep:** dla N=1,2,...: świeży kontener, zamontuj, wykonaj reprezentatywną PACZKĘ operacji
   (mkdir, create, write danych wielo-blokowych, unlink, rename — mix metadanych i danych),
   ustaw `dev.fail_after=N`, wywołaj `gh2_fs_commit` (N-ty zapis paczki commitu failuje =
   symulacja awarii). Porzuć stan w pamięci, **remount świeżo z pliku**, `gh2_fsck` →
   `issues==0` (spójny). Sprawdź, że stan to STARY (paczka niezacommitowana) ALBO NOWY (cała
   paczka) — nigdy częściowy: kanarek (np. zawartość znanego pliku) jest albo stara, albo nowa
   w całości. Zwiększaj N aż commit przejdzie bez awarii (pełne pokrycie punktów zapisu).
2. **Wielokrotny commit + sweep:** kilka paczek z commitami; awaria w środku 3-ciej → stan po
   2-gim commicie (spójny, trwały) — wcześniejsze commity nienaruszone.
3. **Idempotencja remount:** podwójny remount bez zmian → identyczny stan, fsck==0.
4. **fsck wykrywa (negatywne):** sztucznie wstrzyknij niespójność (np. usuń INODE_ITEM zostaw
   DIR_ITEM przez surowy zapis) → gh2_fsck issues>0 (dowód, że fsck faktycznie sprawdza).
5. Regresja v2.0–v2.4 + v1 `0 failed`; ASan czyste.

## Świadome decyzje
- Model awarii = „stop zapisów na punkcie N" (plik-backed; pwrite natychmiast). Testuje, że
  KOLEJNOŚĆ zapisów (SB ostatni) czyni każdy prefiks spójnym — to sedno atomowości CoW.
- gh2_fsck read-only (repair → przyszłość, jak v1 N ale dla v2).
- Pełny group-commit/perf (batch wielu ops) już działa (ops akumulują, commit utrwala).
