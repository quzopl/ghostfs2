# ghostfs v2.7 — Snapshoty (eager refcount): design

**Data:** 2026-06-25
**Część:** v2 (CoW B-tree). Sztandarowa cecha CoW. Wprowadza refcounty bloków odłożone z v2.2.

## Kluczowa decyzja: eager refcount (mark-sweep), NIE btrfs lazy + extent tree
**Refcount(blok) = liczba referencji do niego ze WSZYSTKICH drzew subwolumenów** — czysta
funkcja struktury drzew, **odbudowywalna mark-sweep przy mount** (dokładnie jak mapa wolnej
przestrzeni w v2.2). Konsekwencje:
- **Rdzeń B-drzewa (gh2_btree) NIETKNIĘTY** — 8 mln property-checks zostaje. CoW dalej alokuje
  nowe węzły i „zwalnia" stare przez vtable; zmienia się TYLKO znaczenie „free" → **dekrement**
  (fizyczne zwolnienie dopiero przy refcount 0).
- **BEZ self-referencyjnego drzewa ekstentów** (najtrudniejsza część btrfs) — refcounty w
  pamięci, persystencja niejawna w strukturze drzew, odbudowa mark-sweep.
- **Snapshot = O(drzewo)** inkrement refcount wszystkich osiągalnych bloków (nie O(1) jak
  btrfs lazy) — poprawne, dużo prostsze, akceptowalne dla tego FS. (Lazy = przyszła optymalizacja.)

### Dlaczego to poprawne (dowód szkic)
Po `snapshot`: +1 do KAŻDEGO bloku osiągalnego z fs_root → oryginał i snapshot współdzielą
wszystko (refcount 2). Gdy oryginał CoW-modyfikuje ścieżkę R→I→L: B-drzewo tworzy R'→I'→L'
(refcount 1), „zwalnia" (dekrementuje) skopiowane R,I,L (2→1; snapshot je trzyma). Rodzeństwo
(niemodyfikowane poddrzewa) wskazywane przez R (snapshot) i R' (oryginał) → refcount pozostaje
2 (B-drzewo kopiuje wskaźniki dzieci, nie dotyka ich refcount — i poprawnie, bo teraz 2 rodziców).
Mark-sweep przy mount liczy referencje z obu korzeni → te same refcounty. Niezmiennik:
**refcount(b) == liczba referencji z żywych drzew subwolumenów** — utrzymywany inkrementalnie,
odbudowywalny mark-sweep. To BRAMKA (jak mapa==mark-sweep w v2.2/2.4, teraz refcount==mark-sweep).

## Subwolumeny — drzewo korzeni
`sb.root_tree` przestaje wskazywać bezpośrednio fs_root; wskazuje **drzewo korzeni** (gh2_btree):
itemy `(subvol_id, GH2_ROOT_ITEM, 0) → {fs_root_bptr, flags, name[]}`. Subwolumen domyślny =
id 1 (główny FS). Snapshot = nowy subvol_id → kopia bptr aktualnego fs_root (współdzielenie).
Mount: wczytaj drzewo korzeni; aktywny subwolumen (domyślnie 1) → jego fs_root = aktywne
drzewo FS. Operacje mutują aktywny fs_root (CoW); commit aktualizuje wpis subwolumenu w drzewie
korzeni + superblok. (Migracja: stare kontenery v2 z `root_tree`=fs_root bezpośrednio — format
v2.7 owija w drzewo korzeni; existing v2 containers re-formatowane, zgodnie z polityką „re-create".)

## Refcounty (`src/v2/gh2_space.c` rozszerzenie)
- Stan w pamięci: bitmapa „used" (jest) + **store refcountów** dla bloków o refcount>1 (większość
  ma 1; sparse — hashmap/tablica `{block→count}`, brak wpisu+used == refcount 1). Lub pełna
  tablica u16/u32 per blok (prościej; ~2-4 B/blok). Decyzja w implementacji (prościej = pełna).
- `gh2_ref_inc(block)`, `gh2_ref_dec(block)` (przy 0 → zwolnij w mapie), `gh2_ref_get(block)`.
- `gh2_txn_alloc` free → **dec** (defer do commitu); inc dla snapshotu (defer). Savepoint/
  rollback obejmuje refcount-deltas (undo inkrementów/dekrementów op przy abort).
- Mark-sweep rozszerzony: `gh2_refmap_build_from_roots(dev, &refmap, root_tree)` — dla każdego
  subwolumenu walk drzewa (węzły + bloki danych), zlicz referencje → refcount. used == (rc>0).

## API snapshotów (`src/v2/gh2_fs.c` / vfs / cli)
- `gh2_fs_snapshot(fs, snap_name)` — utwórz subwolumen: nowy id, fs_root = aktualny aktywny
  fs_root, **inc refcount wszystkich bloków osiągalnych z fs_root** (O(drzewo)), dodaj wpis do
  drzewa korzeni; commit. (Snapshot zapisywalny — własny niezależny CoW.)
- `gh2_fs_subvol_list(fs, cb)` — lista subwolumenów (id, name).
- `gh2_fs_subvol_delete(fs, id)` — **dec refcount wszystkich bloków** drzewa subwolumenu
  (free osiągalnych tylko przezeń; współdzielone zostają), usuń wpis z drzewa korzeni; commit.
  Nie usuwaj aktywnego/domyślnego bez przełączenia.
- (Przełączanie aktywnego subwolumenu przy mount: opcja `-o subvol=ID`/CLI — min: dostęp do
  snapshotu read przez ścieżkę `/.snapshots/<name>`? Pierwszy cios: lista + delete + osobny
  mount aktywnego ID. Pełny dostęp do wielu subwolumenów jednocześnie = rozszerzenie.)
- CLI: `snapshot <kontener> <nazwa>`, `subvol-list`, `subvol-del <id>`.

## Testy (`tests/test_v2snap.c`)
1. **Współdzielenie (O(1) miejsca):** zapisz dane, snapshot → liczba zajętych bloków NIE rośnie
   (poza wpisem drzewa korzeni); refcounty bloków drzewa+danych == 2.
2. **Izolacja CoW:** modyfikuj oryginał (write/create/unlink) → snapshot NIEZMIENIONY (zawartość
   przez fs_root snapshotu identyczna jak w chwili snapshotu); modyfikuj „przez snapshot" (jeśli
   zapisywalny) → oryginał niezmieniony.
3. **Refcount == mark-sweep (BRAMKA):** po DOWOLNEJ sekwencji ops+snapshot+delete, in-memory
   refcount == `gh2_refmap_build_from_roots` (bit/licznik-po-liczniku). Brak premature-free
   (dane snapshotu czytelne) ani wycieku (delete zwalnia dokładnie bloki wyłączne).
4. **Delete:** snapshot, modyfikuj oryginał (rozejście), delete snapshot → zwolnione tylko bloki
   wyłączne snapshotu; oryginał + wspólne nietknięte; fsck==0; brak wycieku.
5. **Wiele snapshotów** łańcuch (s1,s2,s3 w różnych momentach) → refcounty poprawne (do 4).
6. **Persystencja + crash:** snapshot+ops+commit+remount → refcounty odbudowane (==mark-sweep),
   wszystkie subwolumeny dostępne; crash-sweep z operacją snapshot → fsck==0, stan stary-albo-nowy.
7. **Regresja:** bez snapshotów wszystko jak v2.0-2.6 (refcount≡1; istniejące testy 0 failed —
   B-drzewo NIETKNIĘTE, gh2_fs zachowanie identyczne przy 1 subwolumenie).
8. ASan/UBSan czyste; gh2_fsck rozszerzony o weryfikację refcount==mark-sweep.

## Świadome ograniczenia / decyzje
- Snapshot O(drzewo) (eager), nie O(1) — prostota+poprawność; lazy = przyszłość.
- Refcounty w pamięci (odbudowa mark-sweep), brak persystentnego drzewa ekstentów — spójne z v2.2.
- Pierwszy cios: snapshoty zapisywalne, lista, delete, dostęp do wybranego subwolumenu. Jednoczesny
  dostęp do wielu (np. /.snapshots/) = rozszerzenie.
- Format: drzewo korzeni owija subwolumeny → re-format kontenerów v2 (polityka „re-create", jak v1→v2).
